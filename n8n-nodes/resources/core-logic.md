## 1. If 節點 (條件判斷)

`If` 節點是工作流中最常用的邏輯組件，用於根據布林條件將資料分為 `true` 和 `false` 兩個路徑。

- **多重條件 (Logical AND/OR)**：您可以在一個 `If` 節點中設定多個條件。
    - **AND**：所有條件都必須滿足，資料才會流向 `true`。
    - **OR**：只要滿足其中一個條件，資料就會流向 `true`。
- **類型比較**：支援 String（字串）、Number（數值）、Boolean（布林）、DateTime（日期時間）等多種數據類型的比較。
- **進階技巧**：利用 `Expression`（表達式）可以進行複雜的邏輯運算，例如 `{{ $json.price > 100 && $json.status === 'active' }}`。

## 2. Switch 節點 (多路徑分流)

當您的邏輯分支超過兩個時，`Switch` 節點是比多個 `If` 節點更優雅的選擇。

- **Routing Rules**：您可以定義多條規則，每條規則對應一個輸出連接埠。
- **匹配模式**：
    - **Equal**：精確匹配。
    - **Regex**：使用正則表達式進行複雜匹配。
    - **Contains/Starts With**：字串包含或開頭匹配。
- **Default Output**：如果沒有任何規則匹配，資料可以流向一個預設的輸出連接埠（可選）。

## 3. Loop Over Items 節點 (循環與批次處理)

舊稱為 `Split in Batches`，此節點用於處理大量資料，防止下游系統過載。

- **Batch Size**：定義每次循環處理的項目數量。
- **循環邏輯**：資料會先進入循環路徑，處理完一組後再回到 `Loop Over Items` 節點領取下一組，直到所有資料處理完畢後，才會流向 `Done` 路徑。
- **應用場景**：API 速率限制（Rate Limiting）、大型資料庫匯入、分段處理長文檔。

## 4. Merge 節點 (資料合併)

`Merge` 節點用於將來自兩個不同分支的資料整合在一起。

- **常用模式**：
    - **Append**：將第二個輸入的資料追加到第一個輸入的資料後面。
    - **Merge By Key**：類似於 SQL 的 `JOIN`，根據共同的欄位（如 `ID`）將兩個來源的欄位合併到同一個項目中。
    - **Multiplex**：笛卡爾積，將第一個輸入的每個項目與第二個輸入的所有項目組合。
    - **Wait for Both**：不合併資料，僅等待兩個輸入都完成後再繼續執行。

## 5. Wait 節點 (執行暫停)

用於在工作流中引入延遲或等待外部事件。

- **Wait Amount**：設定固定的等待時間（秒、分、時、日）。
- **Resume At**：設定一個特定的日期和時間繼續執行。
- **Webhook Resume**：工作流會暫停，直到接收到特定的 Webhook 請求後才繼續。這對於需要「人工審核」或「異步回調」的流程非常有用。

## 6. Stop And Error 節點 (錯誤終止)

當工作流進入不應發生的狀態時，主動拋出錯誤。

- **自定義錯誤訊息**：您可以設定報錯時顯示的文字。
- **工作流狀態**：使用此節點會使整個工作流執行狀態標記為 `Failed`，這對於觸發 `Error Workflow` 非常有用。

## 7. Execute Sub-workflow 節點 (子工作流調用)

將複雜的工作流拆分為多個可重用的模組。

- **封裝邏輯**：將通用的邏輯（如：錯誤通知、資料清洗）寫成子工作流，在多個主工作流中調用。
- **資料傳遞**：可以將主工作流的資料傳遞給子工作流，並接收子工作流返回的處理結果。
